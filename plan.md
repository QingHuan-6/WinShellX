# Windows命令行解释器设计实训计划

## 一、实训目标

本次计算机系统能力实训的题目为“Windows命令行解释器设计”。实训目标是通过设计并实现一个简化版 Windows 命令行解释器，加深对操作系统、系统调用、进程管理、文件系统和控制台程序设计的理解。

本项目计划实现一个运行在 Windows 控制台中的命令解释程序，并命名为 **WinShellX**。程序启动后显示当前目录和提示符 `>`，用户在提示符后输入命令，解释器解析命令并调用相应的 Windows API 完成功能，执行结果直接显示在控制台中。

通过本实训，重点完成以下目标：

1. 理解命令解释程序、系统调用、操作系统内核和硬件之间的层次关系。
2. 学习 Windows API 的基本调用方法。
3. 掌握控制台程序的输入、解析、执行和输出流程。
4. 实现常用内部命令，包括 `cd`、`dir`、`history`、`exit`、`tasklist`、`taskkill`。
5. 完成程序测试、调试和实训报告撰写。

## 二、项目命名与定位

本项目开发的 Shell 命名为 **WinShellX**。

命名含义如下：

1. `Win` 表示程序运行平台为 Windows。
2. `Shell` 表示程序定位为命令行解释器。
3. `X` 表示项目具备可扩展能力，后续可以继续增加新的内部命令、外部命令调用、脚本执行等功能。

WinShellX 的设计定位是一个教学型 Windows 命令行解释器。它不追求完全复刻 `cmd.exe`，而是围绕课程要求，重点展示命令解释器的基本原理、Win32 API 调用方式、模块化设计方法和系统软件开发能力。

## 三、需求分析

### 3.1 功能需求

命令解释器需要支持以下内部命令：

| 命令 | 功能说明 |
| --- | --- |
| `cd <路径>` | 切换当前工作目录 |
| `cls` | 清空控制台屏幕 |
| `dir [路径]` | 显示当前目录或指定目录下的文件、目录信息 |
| `history` | 显示用户在当前控制台会话中输入过的历史命令 |
| `exit` | 退出命令解释器 |
| `tasklist` | 显示系统当前进程信息，包括进程名、PID、线程数等 |
| `taskkill <pid>` | 根据进程 PID 结束指定进程 |
| `help` | 显示当前支持的命令及用法 |
| `alias [name=command]` | 创建、查看命令别名 |
| `unalias <name>` | 删除命令别名 |
| 外部命令 | 当输入不是内部命令时，尝试调用外部程序，例如 `notepad` |

### 3.2 非功能需求

1. 程序应运行在 Windows 控制台环境中。
2. 程序应采用 C 或 C++ 实现，优先使用 C++ 组织代码。
3. 程序应调用 Win32 API 完成文件目录、进程枚举和进程终止等操作。
4. 命令执行后应给出清晰的结果输出或错误提示。
5. 对非法命令、参数缺失、路径不存在、PID 无效等情况应进行基本处理。
6. 代码开发应遵循高内聚、低耦合原则，便于维护和扩展。
7. 新增命令时应尽量只新增独立命令类或命令处理函数，避免修改主循环核心逻辑。

## 四、Windows API 学习计划

本项目重点学习和使用以下 Windows API：

### 4.1 目录与文件相关 API

| API | 用途 |
| --- | --- |
| `GetCurrentDirectory` | 获取当前工作目录 |
| `SetCurrentDirectory` | 设置当前工作目录 |
| `FindFirstFile` | 查找目录中的第一个文件或子目录 |
| `FindNextFile` | 继续查找目录中的文件或子目录 |
| `FindClose` | 关闭文件查找句柄 |

### 4.2 进程管理相关 API

| API | 用途 |
| --- | --- |
| `CreateToolhelp32Snapshot` | 创建系统进程快照 |
| `Process32First` | 获取快照中的第一个进程 |
| `Process32Next` | 获取快照中的下一个进程 |
| `OpenProcess` | 根据 PID 打开进程对象 |
| `TerminateProcess` | 终止指定进程 |
| `CloseHandle` | 关闭系统对象句柄 |

### 4.3 控制台输入输出

控制台输入输出可以使用 C++ 标准库完成，例如 `std::getline`、`std::cout`、`std::cerr`。如需进一步扩展，也可以学习 `ReadConsole`、`WriteConsole` 等 Console API。

## 五、命令解释器总体设计

命令解释器采用循环读取命令的结构。程序启动后进入主循环，每次循环完成一次命令输入、解析和执行。

基本流程如下：

1. 获取当前工作目录。
2. 显示命令提示符，格式为：`当前目录>`。
3. 读取用户输入的一整行命令。
4. 去除输入中的多余空白。
5. 将命令保存到历史记录中。
6. 拆分命令名和参数。
7. 根据命令名调用对应处理函数。
8. 如果不是内部命令，则按类似 `cmd.exe` 的方式解析外部命令：解析首个 token，结合 `PATHEXT` 补全扩展名，在当前目录、系统目录、Windows 目录和 `PATH` 中搜索可执行文件。
9. 输出执行结果或错误提示。
10. 如果命令为 `exit`，结束循环并退出程序。
11. 程序退出时将历史命令保存到 `.winshellx_history` 文件，下次启动时自动加载。
12. 对包含 `>` 的命令执行输出重定向，对包含 `|` 的命令执行单级管道连接。
13. 对以 `&` 结尾的外部命令采用后台方式启动，不等待其退出。
14. 执行命令前检查别名表，将别名展开为真实命令。

交互输入增强流程如下：

1. 用户输入命令前缀时，系统优先根据历史记录显示最近匹配命令作为灰色 hint。
2. 当没有历史匹配时，系统会根据内置命令名或当前路径文件名、目录名生成补全候选。
3. 用户按 `Tab` 可以直接接受当前 hint，实现类似搜索历史和命令行自动补全的效果。
4. 用户按方向键上/下可以在历史命令中前后翻阅。
5. 用户按 `F7` 可以显示最近匹配的历史命令列表，并通过数字选择某一条命令填入输入行。

### 5.1 架构设计原则

WinShellX 的代码开发遵循高内聚、低耦合的设计原则。

1. 主循环只负责流程控制，不直接实现具体命令功能。
2. 命令解析模块只负责字符串解析，不关心命令如何执行。
3. 每个命令模块只负责一个明确功能，例如目录管理、历史记录或进程管理。
4. Windows API 调用集中封装在功能模块中，减少 API 细节在代码中分散出现。
5. 命令注册与命令执行分离，后续新增命令时只需注册新命令处理器。
6. 错误处理统一返回清晰提示，避免每个模块重复编写大量相同逻辑。

建议采用如下抽象方式：

1. 定义 `Command` 或 `ICommand` 接口，统一规定命令名称、帮助信息和执行函数。
2. 使用 `CommandRegistry` 保存命令名与命令处理器的映射关系。
3. 使用 `CommandParser` 将用户输入解析为 `CommandRequest`。
4. 使用 `ShellContext` 保存当前目录、历史记录、输出对象等运行时上下文。
5. 每个命令实现只依赖必要的上下文和参数，不直接控制主循环。

## 六、功能模块划分

### 6.1 主控模块

主控模块负责程序启动、主循环控制、提示符显示和程序退出。它是整个命令解释器的调度中心。

主要职责：

1. 初始化历史命令列表。
2. 循环读取用户输入。
3. 调用命令解析模块。
4. 根据解析结果分发到具体命令模块。

### 6.2 命令解析模块

命令解析模块负责把用户输入的字符串转换为命令名和参数列表。

主要职责：

1. 判断输入是否为空。
2. 提取第一个单词作为命令名。
3. 提取后续内容作为参数。
4. 对命令名进行统一处理，例如转换为小写。
5. 为后续命令执行模块提供结构化数据。

### 6.3 目录命令模块

目录命令模块实现 `cd` 和 `dir`。

`cd` 命令使用 `SetCurrentDirectory` 切换目录，并在失败时输出错误提示。

`dir` 命令使用 `FindFirstFile` 和 `FindNextFile` 遍历目录内容，显示文件名、目录标识、文件大小等信息。

### 6.4 历史命令模块

历史命令模块负责维护当前会话中输入过的命令。

主要职责：

1. 每次用户输入有效命令后保存到列表。
2. 执行 `history` 时按顺序输出命令编号和命令内容。
3. 程序启动时从 `.winshellx_history` 加载历史命令。
4. 程序退出时将历史命令保存到 `.winshellx_history`。
5. 根据当前输入前缀提供历史 hint、命令名补全、路径补全、`Tab` 接受补全、上下键翻阅历史和 `F7` 最近历史选择。

### 6.5 进程管理模块

进程管理模块实现 `tasklist` 和 `taskkill`。

`tasklist` 使用 `CreateToolhelp32Snapshot`、`Process32First` 和 `Process32Next` 枚举系统进程，输出进程名、PID、线程数量等信息。

`taskkill` 根据用户输入的 PID 调用 `OpenProcess` 获取进程句柄，再调用 `TerminateProcess` 结束进程。若 PID 不存在、权限不足或参数非法，应输出错误提示。

### 6.6 错误处理模块

错误处理模块负责统一处理常见错误情况。

常见错误包括：

1. 未知命令。
2. 命令参数缺失。
3. 路径不存在或不可访问。
4. PID 不是合法数字。
5. 打开进程失败。
6. 终止进程失败。
7. 外部程序启动失败。

### 6.7 外部命令执行模块

外部命令执行模块负责处理内部命令表中不存在的输入。

主要职责：

1. 解析用户输入的第一个 token 作为外部命令名，其余内容作为参数。
2. 若命令没有扩展名，则读取 `PATHEXT`，按扩展名顺序生成 `.COM`、`.EXE`、`.BAT`、`.CMD` 等候选。
3. 若命令没有目录部分，则依次在当前目录、系统目录、Windows 目录和 `PATH` 环境变量目录中搜索候选程序。
4. 对 `.exe`、`.com` 等可执行文件直接调用 `CreateProcess`。
5. 对 `.bat`、`.cmd` 批处理文件使用 `cmd.exe /c` 包装执行，符合 Windows 批处理执行方式。
6. 支持带参数的外部命令，例如 `notepad test.txt`。
7. 等待外部程序执行结束后返回 WinShellX。
8. 外部程序不存在或启动失败时输出清晰错误信息。

### 6.8 管道与重定向模块

管道与重定向模块负责处理命令中的 `>` 和 `|`。

主要职责：

1. 支持 `dir > out.txt`，将命令输出写入文件。
2. 支持单级管道，例如 `dir | find "cpp"`。
3. 支持管道结果再重定向，例如 `dir | find "cpp" > result.txt`。
4. 对内部命令通过临时重定向 `std::cout` 捕获输出。
5. 对外部命令通过 Windows 匿名管道连接标准输入和标准输出。

### 6.9 别名、后台任务与彩色输出模块

主要职责：

1. 支持 `alias ll=dir`、`alias`、`alias ll` 和 `unalias ll`。
2. 别名保存到 `.winshellx_aliases`，下次启动自动加载。
3. 支持外部命令后台执行，例如 `notepad &`，启动后立即返回 WinShellX 并显示 PID。
4. 使用 `SetConsoleTextAttribute` 为提示符、帮助标题、目录名和成功提示提供彩色输出。

## 八、代码规范与扩展规范

### 8.1 代码规范

1. 文件名和类名保持一致，例如 `CdCommand.h` 对应 `CdCommand`。
2. 类名使用大驼峰命名法，例如 `CommandParser`。
3. 函数名使用小驼峰命名法，例如 `parseInput`、`executeCommand`。
4. 常量使用全大写或 `constexpr` 命名，例如 `MAX_HISTORY_SIZE`。
5. 每个函数只完成一个明确任务，避免过长函数。
6. 对 Windows API 返回值必须进行检查，失败时输出清晰错误信息。
7. 对用户输入必须进行参数校验，避免空参数、非法 PID、非法路径导致程序崩溃。
8. 公共接口写必要注释，复杂逻辑处添加简短说明。

### 8.2 扩展规范

新增一个内部命令时，按以下步骤完成：

1. 在 `include/commands/` 中新增命令头文件。
2. 在 `src/commands/` 中新增命令实现文件。
3. 实现统一的命令接口，包括命令名称、帮助信息和执行函数。
4. 在 `CommandRegistry` 中注册命令。
5. 在 `tests/` 中补充对应测试用例。
6. 在 `README.md` 或帮助命令中补充命令说明。

通过该方式，新增命令不需要修改主循环，也不需要影响已有命令模块，体现低耦合和可扩展设计。

## 十、测试计划

测试分为自动化测试和手工测试两类。自动化测试主要验证命令解析、参数校验和部分不依赖系统状态的模块逻辑；手工测试主要验证目录切换、进程枚举、进程终止等与 Windows 系统环境相关的功能。

### 10.1 自动化测试规划

| 测试文件 | 测试重点 |
| --- | --- |
| `parser_tests.cpp` | 测试空输入、普通命令、带参数命令、带空格路径和带引号路径 |
| `command_tests.cpp` | 测试命令注册、未知命令处理、参数数量校验 |
| `manual_test_cases.md` | 记录需要人工执行的功能测试步骤和结果 |

自动化测试重点：

2. `CommandRegistry` 能正确注册和查找命令。
3. 非法命令能返回统一错误提示。

6. 历史命令文件能在程序退出后保存，并在下次启动时加载。

## 十一、可扩展亮点与加分点

在完成基础要求后，WinShellX 可以加入以下扩展功能作为答辩亮点：

1. `help` 命令：显示所有支持命令及用法，使程序更接近真实 Shell。
2. `cls` 命令：调用 Windows Console API 清空控制台屏幕，改善交互体验。
3. 外部命令执行：当输入不是内部命令时，尝试通过 `CreateProcess` 启动外部程序。
4. 管道或重定向雏形：支持将命令输出重定向到文件，例如 `dir > out.txt`。
5. 历史命令持久化与搜索式交互：退出程序前将历史记录保存到文件，下次启动后加载；输入前缀时显示 hint，支持 `Tab` 补全和 `F7` 最近历史选择。
6. 智能补全：根据最近历史、内置命令名、当前目录文件名或目录名补全输入内容。
7. 彩色输出：目录、普通文件、错误信息使用不同颜色显示，提高可读性。
8. 宽字符支持：使用 `W` 版本 Win32 API，增强中文路径兼容性。
9. 插件式命令扩展：将命令注册机制设计成可扩展结构，后续可以独立添加更多命令。
10. 更完整的 `dir` 信息：显示文件修改时间、文件大小统计和磁盘空间信息。

当前已经实现的加分点包括 `help` 命令、`cls` 清屏命令、外部命令执行、输出重定向、单级管道、后台任务 `&`、命令别名、彩色输出、历史命令持久化、历史 hint、命令名补全、路径补全、`Tab` 接受补全、上下键翻阅历史和 `F7` 最近历史选择。后续可继续扩展多级管道、环境变量展开和更完整的批处理语法。

## 十四、风险与解决方案

| 风险 | 可能原因 | 解决方案 |
| --- | --- | --- |
| Windows API 调用失败 | 参数错误或权限不足 | 查阅官方文档，检查返回值并使用错误码定位问题 |
| `taskkill` 无法结束部分进程 | 权限不足或系统保护进程 | 给出明确错误提示，不强制结束系统关键进程 |
| 路径中包含空格导致解析错误 | 命令解析不完善 | 支持带引号路径或将 `cd`、`dir` 后续内容整体作为路径参数 |
| 中文路径显示异常 | 控制台编码不一致 | 根据开发环境设置控制台编码，必要时使用宽字符 API |
| 输出格式混乱 | 信息字段宽度不统一 | 对文件列表和进程列表进行简单对齐 |
| 测试不充分 | 只测试正常输入 | 增加非法路径、非法 PID、空命令等异常测试 |
